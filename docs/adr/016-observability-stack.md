# ADR 016: Observability Stack — VictoriaMetrics + Grafana

**Status:** Implemented
**Date:** 2026-05-09
**Deciders:** Project maintainers

## Context

The homelab had zero observability (issue #13): no metrics, no logs, no traces. Production issues were discovered reactively via SSH or user reports. Certificate expiry, resource exhaustion, and application crashes were invisible.

The classic solution — `kube-prometheus-stack` — is prohibitively heavy for a single-node homelab (several GB RAM, aggressive disk usage from full Prometheus server + Alertmanager + bundled exporters).

## Decision

Deploy a lightweight observability stack:

- **VictoriaMetrics (single-node)** as the metrics backend, replacing Prometheus
- **Grafana** (standalone chart, not bundled) for dashboards
- **prometheus-node-exporter** (DaemonSet) for host-level metrics
- **kube-state-metrics** for Kubernetes object metrics

All deployed via Helm into a dedicated `observability` namespace.

## Rationale

### Why VictoriaMetrics over Prometheus?

- 4–7× lower RAM and CPU under comparable load
- Native Prometheus scrape protocol — all existing `/metrics` endpoints work unchanged
- Built-in scraping (no separate scrape operator needed for single-node)
- PromQL-compatible query API — Grafana data source config requires zero changes vs Prometheus
- Single binary, no side-cars or sharding required at homelab scale

### Why not kube-prometheus-stack?

- Bundles Prometheus server (memory-hungry), Alertmanager, and many exporters as a monolith
- Designed for production multi-node clusters; overkill for a homelab
- Resource usage grows aggressively with scrape interval and retention

### Why not Netdata or Coroot?

- **Netdata**: excellent for per-node real-time monitoring, but cross-node PromQL queries are awkward and custom dashboards are limited
- **Coroot**: promising eBPF-based all-in-one, but too early-stage for this setup

### Why separate Grafana instead of vmui?

- `vmui` (VictoriaMetrics built-in UI) is a PromQL explorer, not a dashboard builder
- Grafana provides pre-built Kubernetes dashboards, alerting, and a familiar workflow
- Standalone Grafana chart is lightweight (~150MB RAM) when not bundled with the full Prometheus stack

## Architecture

```
Cluster nodes
  └── node-exporter (DaemonSet) → /metrics:9100
k8s API
  └── kube-state-metrics → /metrics:8080

VictoriaMetrics (single pod)
  ├── scrapes node-exporter + kube-state-metrics + kubelet + cAdvisor
  ├── stores metrics (10Gi, longhorn-uncritical, 3-month retention)
  └── exposes PromQL API on :8428

Grafana (single pod)
  ├── datasource: VictoriaMetrics (Prometheus-compatible)
  ├── persistent dashboards (1Gi, longhorn-uncritical)
  └── exposed via HTTPRoute → Traefik Gateway → Cloudflare Tunnel
```

## Consequences

### Positive

- Dramatically lower resource footprint vs kube-prometheus-stack
- Drop-in compatible with all Prometheus `/metrics` endpoints already in the stack (Traefik, etc.)
- Full PromQL support for cross-node and cross-service queries
- Grafana provides polished dashboards and alerting

### Negative

- No built-in log aggregation (Loki would need to be added separately)
- No distributed tracing (Tempo would need to be added separately)
- Grafana admin password must be retrieved from the auto-generated Secret on first deployment

## Alternatives Considered

| Option | Reason rejected |
|---|---|
| kube-prometheus-stack | Too heavy (several GB RAM) |
| Netdata alone | Poor cross-node PromQL, limited custom dashboards |
| OpenObserve | Metrics support secondary; log-first tool |
| Coroot | Promising but too early-stage |
| VictoriaMetrics + vmui | vmui is not a full dashboard builder |

## Implementation Notes

- Grafana admin password is auto-generated by the Helm chart and stored in the `grafana` Secret in the `observability` namespace. Retrieve with:
  ```
  kubectl get secret -n observability grafana -o jsonpath="{.data.admin-password}" | base64 -d
  ```
- VictoriaMetrics scrape config uses Kubernetes pod/service annotation discovery (`prometheus.io/scrape: "true"`) plus static targets for kubelet and cAdvisor
- Storage uses `longhorn-uncritical` (no backup — metrics data is ephemeral by nature)
